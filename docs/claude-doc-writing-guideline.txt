# 작성 지침

이 프로젝트의 목적은 **클로드코드 설계서를 입력으로 받아, Claude Code가 실제 구현·수정·테스트·검증 작업을 수행할 때 직접 참조할 수 있는 `CLAUDE.md` 파일을 만드는 것**이다. 산출물은 소개문, README, 요약문이 아니라 **Claude Code의 행동 기준과 의사결정 기준을 고정하는 운영 문서**여야 한다.

## 최우선 절대 원칙

1. 입력된 프로젝트 설계서 내용을 `CLAUDE.md` 최적화 문서로 만드는 과정에서 **절대 누락되는 요소가 있어서는 안 된다.**
2. `CLAUDE.md` 파일 크기 최소화 조건 때문에 누락이 생길 것 같다면, **참조 문서를 최대한 많이 만들어서라도 절대 누락되는 부분이 없어야 한다. 참조문서 설명은 맨뒤에 한번에 정리하는게 아니라 각 카테고리에 넣어놓는다.**
3. 참조문서의 경우 Agent, Skills 문서등, 클로드 코드가 프로젝트 구성 과정에서 자동으로 생성하는 파일이 아니라, docs등의 폴더에 별도의 MD파일을 만들어서 구성한다.
4. Agent, Skills 문서등, 클로드 코드가 프로젝트 구성 과정에서 자동으로 생성하는 파일은, 프로젝트 생성과정에서 클로드코드가 생성한다.
5. Claude용 문서를 생성한 이후에는, **입력받은 원본 프로젝트 설계서 대비 누락된 부분이 0%인지 반드시 더블체크해야 한다.**
6. 프로젝트 설계서 안에 **LLM이 처리할 부분과 스크립트로 처리할 부분을 분리해 놓은 항목이 있다면, 이 항목은 반드시 최종 `CLAUDE.md` 본문에 존재해야 한다.**
7. README파일은 따로 만들지 말고, README에 들어가야만 하는 내용들은 다른 문서에 포함시킨다.
8. 코드나 문서 수정 후 QA 에이전트가 반드시 검사를 해야만 함.
9. QA 전용 별도 소프트웨어 금지: QA는 사용자와 동일한 파이프라인을 직접 수행해야 하며, 사용자가 받게 되는 결과물을 가지고 판단을 해야 함
10. "AI 판정은 **현재 실행 중인 Claude Code 세션(대화)**이 직접   
  수행하며, 별도로 anthropic 패키지를      
  호출하거나 API 키를 설정할 필요가 없다."

`CLAUDE.md`는 사람에게 프로젝트를 설명하는 문서가 아니라, Claude Code가 **무엇을 해야 하는지, 무엇을 하지 말아야 하는지, 무엇을 우선해야 하는지, 어떤 기준으로 완료를 판단해야 하는지**를 빠르게 이해하도록 만드는 작업 기준 문서다. 따라서 설계서를 자연스럽게 요약하는 방식으로 작성하면 안 된다. 설계서를 분석해서 **행동 가능한 규칙, 고정 제약, 우선순위, 금지사항, 테스트 기준, 완료 기준**으로 재구성해야 한다.

가장 중요한 원칙은 **추상적인 설명보다 실행 가능한 규칙을 우선하는 것**이다.
“적절히 처리한다”, “합리적으로 개선한다”, “사용성을 고려한다”, “필요시 구조를 정리한다” 같은 표현은 최대한 사용하지 않는다. 대신 다음처럼 바로 행동으로 옮길 수 있는 문장을 사용한다.

* 자동 감지 신뢰도가 낮으면 자동 확정하지 말고 사용자 확인 단계로 넘긴다.
* 출력 헤더명과 열 순서는 명시적 요구 없이는 변경하지 않는다.
* 기존 구조를 유지하면서 필요한 파일만 최소 범위로 수정한다.
* 테스트 없이 핵심 로직을 교체하지 않는다.

이 프로젝트는 항상 **두 단계로 분리해서 사고해야 한다.**

### 1단계: 설계서 분석

설계서에 들어 있는 프로젝트 정의, 목표, 비목표, 우선순위, 입력 계약, 출력 계약, 워크플로우, 예외 케이스, 실패 정책, 테스트 기준, 완료 기준, LLM/스크립트 역할 분리, 제약 조건, 도메인 규칙을 빠짐없이 파악해야 한다. 설계서가 추상적이면 그대로 옮기지 말고, Claude Code가 바로 실행할 수 있는 규칙으로 바꿔야 한다. 이 단계의 목표는 **요약이 아니라 원본 설계서 정보 누락 0% 확보**다.

### 2단계: 최종 `CLAUDE.md` 작성

최종 `CLAUDE.md`에는 Claude Code가 매 세션마다 반드시 알아야 하는 내용만 남긴다. 예를 들면 프로젝트 핵심 목적, 절대 바뀌면 안 되는 입출력 계약, 수정 우선순위, 빌드/테스트 명령, 자주 틀리는 함정, 금지사항, 완료 기준, LLM 처리 영역과 스크립트 처리 영역 분리 같은 내용이다. 긴 배경 설명, 장문의 예시, 자주 바뀌는 참고 정보, 특정 작업에서만 필요한 세부 절차는 별도 문서로 분리한다. 단, 분리되더라도 **전체 문서 구조 기준으로 원본 설계서 정보는 100% 보존되어야 한다.**

최종 `CLAUDE.md`는 다음 성격을 가져야 한다.

* 사람 대상 설명문이 아니라 Claude Code 대상 작업 기준서일 것
* 장황한 설명보다 행동 규칙이 우선일 것
* 코드만 읽어서는 알기 어려운 지속적 규칙을 담을 것
* 프로젝트 전반에 항상 적용되는 규칙만 남길 것
* 짧고 밀도 높게 유지할 것
* 긴 참고 내용은 분리하되 정보 손실은 0%일 것

`CLAUDE.md`를 생성할 때는 다음 항목을 우선 추출하고 압축해야 한다.

1. **프로젝트 정의**: 무엇을 입력으로 받고, 어떤 처리를 거쳐, 무엇을 출력하는지 한두 문장으로 고정한다.
2. **비목표와 범위 밖 항목**: 로그인, 관리자 기능, 추천 시스템, 전면 리팩토링, UI 재설계, 범위 밖 자동화 등 포함되지 않는 항목을 명시한다.
3. **우선순위**: 정확성, 기존 동작 유지, 출력 형식 준수, 사용자 수정 가능성, 예외 처리 안정성, 성능, 코드 미관 등 무엇을 먼저 지킬지 순서대로 적는다.
4. **입력 계약**: 허용 형식, 필수/선택 필드, 헤더 위치 가능성, 예외 상황, 인코딩, 크기 제한 등을 구체적으로 적는다.
5. **출력 계약**: 출력 형식, 시트 구조, 헤더명, 열 순서, 파일명 규칙, 정렬 방식, 빈 행 처리, 다운로드 방식을 구체적으로 적는다.
6. **단계별 워크플로우**: 입력 수집 → 유효성 검사 → 파싱 → 구조 분석 → 후보 탐지 → 사용자 확인 → 정규화 → 변환 → 검증 → 출력 생성 같은 흐름을 고정한다.
7. **판단과 코드의 역할 분리**: 무엇을 스크립트로 처리할지, 무엇을 규칙 기반으로 처리할지, 무엇을 LLM이 처리할지, 무엇을 사용자 확인 대상으로 남길지 분리한다. 이 항목은 설계서에 존재하면 반드시 최종 `CLAUDE.md` 본문에 포함해야 한다.
8. **수정 원칙**: 기존 구조 유지, 최소 범위 수정, 구조 개편 제한, 미관용 리팩토링 후순위 원칙을 명시한다.
9. **금지사항**: UI 재설계 금지, 출력 포맷 임의 변경 금지, 검증되지 않은 라이브러리 추가 금지, 테스트 없는 핵심 로직 교체 금지, 예외를 무조건 숨기는 처리 금지 등을 적는다.
10. **빌드·실행·테스트 명령**: 설치, 실행, 린트, 테스트, 빌드 명령을 짧고 정확하게 적는다.
11. **예외 케이스와 함정**: 헤더가 첫 줄이 아닐 수 있음, 공백 행, 병합 셀, 문자열 숫자, 열 순서 고정, 자동 감지 실패 시 사용자 확인 필요 같은 핵심 함정을 적는다.
12. **테스트 기준과 완료 기준**: 형식 계약 만족, 핵심 예외 케이스 통과, 기존 정상 케이스 유지, 지정된 테스트 명령 성공, 원본 설계서 대비 누락 0% 검토까지 포함한다.

`CLAUDE.md` 파일의 길이를 줄이기 위해 별도 문서를 만들고 참조만 하도록 구성하는 경우, 이 프로젝트는 `CLAUDE.md` 본문만 생성하고 끝내면 안 된다. 최종 결과물에는 `CLAUDE.md`에서 참조하는 **모든 보조 문서도 함께 포함**되어야 한다. 즉, ChatGPT는 `CLAUDE.md`를 짧게 유지하기 위해 분리한 문서가 있다면, 해당 문서들의 제목, 역할, 참조 구조, 실제 본문 초안까지 함께 작성해야 하며, 사용자가 추가로 다시 요청하지 않아도 **참조 구조 전체가 완성된 상태**로 제공해야 한다.

분리 대상 문서는 예를 들어 다음과 같을 수 있다.

* 상세 워크플로우 문서
* 예외 케이스 및 실패 정책 문서
* 입력/출력 계약 상세 문서
* 테스트 케이스 및 완료 기준 문서
* 도메인 규칙, 용어 사전, 매핑 규칙 문서
* LLM 처리 영역과 스크립트 처리 영역을 분리한 상세 규칙 문서
* 특정 디렉토리 또는 특정 파일 유형에만 적용되는 `.claude/rules/` 문서
* `@import`로 참조할 운영 규칙 문서
* 반복 절차를 담은 skills 문서

문서를 분리할 때는 다음 원칙을 반드시 따른다.

* `CLAUDE.md`에 남길 핵심 규칙과 별도 문서로 분리할 내용을 명확히 구분한다.
* 참조 경로와 문서 역할이 `CLAUDE.md` 안에서 분명히 드러나야 한다.
* 참조 대상 문서는 이름만 제안하지 말고 실제 본문까지 함께 작성한다.
* 최종 산출물은 필요 시 **`CLAUDE.md` + 보조 문서 세트** 형태로 완성되어야 한다.
* 사용자가 다시 요청하지 않아도 참조 문서까지 자동으로 함께 작성한다.
* 보조 문서는 축약판이 아니라, 생략된 세부 규칙을 실제 작업에 쓸 수 있을 정도로 구체적으로 담아야 한다.
* 문서를 분리하더라도 역할 중복과 충돌을 최소화해야 한다.
* 별도 문서까지 포함한 전체 구조를 기준으로 **원본 설계서 대비 누락 0%**를 검증해야 한다.

이 프로젝트에서 답변을 작성할 때는 항상 아래를 점검해야 한다.

* 이 문장은 Claude Code가 읽자마자 바로 행동으로 옮길 수 있는가
* 이 규칙은 프로젝트 전반에 항상 적용되는가
* 이 내용은 코드만 읽어서는 알기 어려운가
* 이 내용은 매 세션마다 알아야 할 만큼 중요한가
* 이 내용을 최종 `CLAUDE.md`에 넣는 것이 적절한가, 아니면 별도 문서로 분리하는 것이 더 적절한가
* 이 문장이 실제 구현·수정·검증 기준으로 작동하는가
* 이 규칙이 다른 규칙과 충돌하지 않는가
* 길이를 줄이면서도 핵심 제약을 잃지 않았는가
* 원본 설계서의 항목이 단 하나라도 누락되지 않았는가
* `CLAUDE.md` 본문과 보조 문서를 합친 전체 구조가 원본 설계서를 100% 커버하는가
* LLM 처리 영역과 스크립트 처리 영역 분리 항목이 최종 `CLAUDE.md`에 포함되었는가
* 최종 생성 후 원본 설계서와 대조하여 누락률 0%를 더블체크했는가

이 프로젝트는 설계서를 그대로 재서술하는 작업을 하면 안 된다. 설계서를 분석해서, **Claude Code가 항상 알아야 하는 규칙**만 추려 짧고 강한 `CLAUDE.md`로 만들고, 나머지 긴 참고 정보는 별도 문서로 분리해야 한다. 최종 목표는 “설계서를 예쁘게 요약한 문서”가 아니라, **Claude Code가 실제로 더 정확하고 일관되게 움직이게 만드는 운영 문서**를 만드는 것이다.
